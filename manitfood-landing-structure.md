# Manitfood — структура лендинга (спека для вёрстки и дизайн-системы)

> Документ описывает **информационную архитектуру и воронку** лендинга.
> Не фиксирует финальный текст, фотографии и цвета — они прорабатываются отдельно.
> Задача документа: (1) по нему собирается вёрстка, (2) из него выводится дизайн-система.

---

## 1. Ядро позиционирования

**Продукт:** Telegram-бот, который превращает фото блюда с телефона в профессиональное.

**Единое обещание (для всех):** «Фото блюда с телефона — результат как из фотостудии. Без фотографа, реквизита и ожидания.»

**Задача лендинга:** продажа сервиса → целевое действие = открыть бота. Всё подчинено воронке.

---

## 2. Две независимые оси (ключевая модель)

В продукте **две ортогональные оси**. Их нельзя приравнивать друг к другу (прошлая версия лендинга это делала — отсюда и «два параллельных лендинга»).

**Ось «КТО» — сегмент клиента:**
- **Заведение / бизнес** — кафе, рестораны, доставка, крупный кейтеринг. Боль: съёмка у фотографа дорогая (3 000–15 000 ₽), меню постоянно меняется (сезонка), заказанная съёмка быстро устаревает.
- **Дом / малый бренд** — домашние кондитеры, небольшие цеха, частный кейтеринг. Боль: ведут соцсети и постят ежедневно, но дома нет хорошего света/композиции; при этом студийный глянец отпугивает — покупатель должен узнать именно то блюдо, что закажет.

**Ось «ЧТО НУЖНО» — режим бота (= две задачи, две модели оплаты):**
- **«Меню»** — эталонные фото пакетами позиций, печатное качество, каталог/витрина. Дороже за кадр, покупается обдуманно.
- **«Поток»** — подписка, ежедневный контент для ленты, дешевле за кадр. Позиционируем как **другую задачу** (скорость и объём, «настоящий» вид), а не как «качество похуже».

**Матрица (обе оси независимы — любому сегменту нужны оба режима):**

|                       | Режим «Меню»                          | Режим «Поток»                         |
|-----------------------|---------------------------------------|---------------------------------------|
| **Заведение**         | Меню и карточки блюд для печати/сайта  | Соцсети заведения на постоянке         |
| **Дом / малый бренд** | Портфолио тортов/позиций как «меню»    | Ежедневные сторис и посты, Авито       |

**Следствие для структуры:** разводим людей по **сегменту** (эмоция, «это про меня»), но сводим всех в **один** блок по режимам и тарифам.

## 3. Принцип структуры: расхождение → схождение (diverge → converge)

```
        ┌──────────────────────────────┐
        │  1. HERO — общее обещание     │  ← для всех
        │  2. ЖИВОЕ ДО/ПОСЛЕ (вау)      │
        └──────────────┬───────────────┘
                       │  расхождение
        ┌──────────────▼───────────────┐
        │  3. Развилка «Что вы делаете?»│
        │  4. Сегмент-зона (морфинг):   │
        │     Заведение  ⇄  Дом/бренд   │
        └──────────────┬───────────────┘
                       │  схождение (возврат в master-мир)
        ┌──────────────▼───────────────┐
        │  5. Как это работает          │  ← снова для всех
        │  6. Режимы и тарифы (единый)  │
        │  7. FAQ / возражения          │
        │  8. Финальный CTA → бот       │
        └──────────────────────────────┘
```

Эта форма (развилка → возврат в общее русло) — сама по себе анти-нейрослоп: большинство ИИ-лендингов это плоская стопка одинаковых секций.

## 4. Тематизация: master-мир + две сегмент-подтемы

- **Master-мир** — нейтральная фирменная тема. В ней живут все общие секции (0, 1, 2, 5, 6, 7, 8, 9). Это **ядро дизайн-системы**.
- **Сегмент-подтемы** — «тёмная студия» (заведение) и «тёплый дом» (дом/бренд). Живут **только в блоке 4** и переключаются морфингом на месте.
- **Эхо акцента (решено):** акцентные элементы общих блоков — кнопки, акцентные тексты, ссылки, подчёркивания — берут акцент выбранного сегмента. Структура и фоны/поверхности общих секций (тарифы, FAQ, «как работает») остаются нейтральными master-миром — иначе визуально ре-склеиваются оси (см. §2). То есть морфится «цвет акцента и примеры», а не вся тема секции.

Почему не «весь сайт морфит»: (1) перекрас тарифов/FAQ подразумевал бы, что предложение зависит от сегмента — оно не зависит; (2) это удвоило бы дизайн-систему; (3) исчез бы момент «схождения».

## 5. Посекционная карта

Для каждой секции: **цель** (стадия воронки) · **содержание** · **ключевое сообщение** · **действие** · **мир/тема** · **заметки/состояния**.

### 0. Sticky-хедер
- **Цель:** навигация + постоянный вход в воронку.
- **Содержание:** лого · якоря «Как работает · Тарифы · Вопросы» · одна primary-CTA.
- **Сообщение:** —
- **Действие:** CTA «Попробовать бесплатно» (единый сквозной, см. §6.1).
- **Мир:** master.
- **Заметки:** якоря дают прыжки по странице (чинит «нет навигации»). На мобиле — компактно, CTA всегда видима.

### 1. Hero
- **Цель:** Внимание / обещание. Первый экран для обоих сегментов сразу.
- **Содержание:** заголовок-обещание, подзаголовок, primary-CTA + «первое фото бесплатно, без карты».
- **Сообщение:** «Фото с телефона → как из студии».
- **Действие:** primary-CTA → в бота; вторичная кнопка — скролл к §2 с честной подписью «Посмотреть как» (не «Открыть бота»).
- **Мир:** master (нейтральный, до развилки).
- **Визуал (акцентный якорь первого экрана):**
  - **Рекомендация — мокап телефона с Telegram-ботом:** экран с реальным диалогом бота, где в чат «прилетает» готовое фото (после). Это сразу заземляет продукт (это буквально Telegram-бот), делает первый экран продукт-специфичным (лечит замечание «шаблонный каркас») и современным.
  - **Фон:** мягкий акцентный слой (тёплый градиент / деликатная фактура / световое пятно-спот), НЕ перегруженное фото — заголовок должен оставаться читаемым.
  - Финальный выбор мокап vs. другой акцентный герой-элемент фиксируется на этапе дизайн-системы; каркас секции держит оба варианта (текст-колонка + визуал-колонка).
- **Заметки:** здесь ещё не выбираем сегмент — крючок универсальный.

### 2. Живое «до/после»
- **Цель:** Интерес / доверие к «магии». Главный эмоциональный WOW-момент.
- **Содержание:** большое фото **ДО** на весь экран (full-bleed); по мере скролла оно кинематографично сменяется на **ПОСЛЕ** (reveal/wipe сверху вниз или по маске). Явные подписи «обычное фото» ↔ «результат Manitfood». Один сильный пример, поданный красиво, — не галерея.
- **Сообщение:** «Разница видна за секунду».
- **Действие:** мягкий переход к развилке.
- **Мир:** master (кинематографичный, тёмный фон допустим как приём).
- **Заметки / состояния:**
  - Нужна РЕАЛЬНО контрастная пара (телефонный снимок ↔ студийный результат) — сейчас плейсхолдер, заменяется позже.
  - Обязателен no-JS fallback (статичная пара side-by-side), иначе секция «залипает» на «до».
  - Дисклеймер про иллюстративность — заметный, не полупрозрачный мелкий текст.
  - Учесть `prefers-reduced-motion`: для чувствительных к движению — без скролл-джекинга, простой показ пары.
  - Отличие от §5: здесь ОДИН кинематографичный пример (эмоция), в §5 — интерактивный конфигуратор из многих (рацио). Не дублировать.

### 3. Развилка «Что вы делаете?»
- **Цель:** Релевантность / самоидентификация. Точка расхождения.
- **Содержание:** заголовок-вопрос + **две крупные интерактивные карточки** (не базовые кнопки), формулировки по ситуации/задаче:
  - «У меня заведение / бизнес» (кафе, ресторан, доставка, крупный кейтеринг)
  - «Я готовлю сам / небольшой бренд» (домашние кондитеры, цеха, частный кейтеринг)
- **Оформление карточек:** каждая — с сегмент-конгруэнтным фото (ресторанная подача ↔ домашний торт), затемняющий оверлей и подпись. При наведении — плавный zoom фото + усиление оверлея; выбранная карточка активна и подсвечена акцентом сегмента. Современно, крупно, «дорого».
- **Сообщение:** «Покажем, как это работает именно для вас».
- **Действие:** выбор переключает сегмент-зону (§4) морфингом на месте.
- **Мир:** переход master → сегмент-подтема.
- **Заметки:**
  - **Дефолт — «Заведение» (кафе/рестораны)** preselected, чтобы блок не был пустым.
  - Кейтеринг сознательно попадает в оба — формулировки по задаче, а не по жёсткому «кто вы». **Третьего входа под кейтеринг нет** (решено).
  - Опционально: hover по карточке даёт микро-превью соответствующего мира сегмента (усиливает связь «выбор → морфинг»).

### 4. Сегмент-зона (морфинг: Заведение ⇄ Дом/бренд)
- **Цель:** Желание. Эмоция под конкретный сегмент.
- **Содержание (единый каркас, наполнение морфит):**
  1. **Боль сегмента** (заведение: дорогой фотограф + меняющееся меню; дом: соцсети не продают, но глянец отпугнёт).
  2. **Обещание под сегмент** (заведение: студийный результат из телефона за минуты, дешевле съёмки; дом: настоящее блюдо, но со светом/композицией «как из портфолио»).
  3. **1 сильный пруф** под сегмент (пример «до/после» релевантный именно этой аудитории — торт для дома, блюдо меню для заведения).
- **Сообщение:** «Это решает именно вашу задачу».
- **Действие:** мягкий переход к §5 (НЕ отдельная цена/CTA-воронка внутри сегмента — это уводило бы в дубли).
- **Мир:** сегмент-подтема (тёмная студия / тёплый дом), морфинг с анимацией.
- **Заметки:**
  - Каркас одинаковый для обоих → морфится контент+тема, структура не дублируется.
  - Не встраивать сюда тарифы (раньше тут был дубль превью «Потока» — убрать).
  - Морфинг — смысловой момент, здесь уместна заметная анимация перехода.

### 5. Как это работает — интерактивный конфигуратор
- **Цель:** Понимание / снятие сложности + рациональный WOW («я сам собрал результат»). Первая общая секция после схождения.
- **Содержание:** 3 шага с переключаемыми примерами; по мере выбора справа (или в центре) обновляется **готовое изображение**. Пользователь буквально собирает результат:
  1. **Шаг 1 — «Загрузите снимок»:** набор из ~5 типичных «ДО» (бургер, торт, салат и т.д. — те, что реально шлют пользователи). Переключаются кнопками/превью.
  2. **Шаг 2 — «Выберите стиль»:** переключаемые варианты стиля (берём самый наглядный кадр под каждый стиль).
  3. **Шаг 3 — «Локация / фон»:** переключаемые варианты фона/локации.
  - Итог: на каждую собранную комбинацию (блюдо × стиль × локация) показываем **заранее сгенерированное** готовое фото.
- **Сообщение:** «Три шага, пара минут — и вы управляете результатом».
- **Действие:** под конфигуратором — CTA «Собрать своё в боте» → в бота.
- **Мир:** master.
- **Заметки / состояния (важно для продакшена):**
  - **Комбинаторика:** число заранее генерируемых картинок = блюда × стили × локации. Держать наборы небольшими и фиксированными (напр. 5 × 4 × 3 = 60), иначе объём генерации и вес страницы взрываются. Точные числа зафиксировать до генерации ассетов.
  - Все комбинации — статичные предгенерированные изображения (не онлайн-генерация). Нужны **предзагрузка/кеш** и плавный кроссфейд при переключении, без «прыжков».
  - Дефолтная комбинация показывается сразу при загрузке секции (не пустой холст).
  - Этот блок **заменяет отдельную галерею стилей** — вариативность стилей демонстрируется здесь (шаг 2). При желании — ссылка «все стили».
  - Мобайл: шаги вертикально, превью-переключатели — горизонтальный скролл; тач-таргеты ≥44px; ленивая загрузка неактивных комбинаций.
  - Fallback без JS: показать одну готовую тройку «ДО → стиль → результат» статикой.

### 6. Режимы и тарифы (единый блок — единственный источник правды)
- **Цель:** Решение / снятие ценового возражения.
- **Содержание:** табы **«Меню»** и **«Поток (подписка)»**. Каждый таб = короткое описание задачи режима + его тарифы. Плашка «первое фото бесплатно, без карты · отмена в любой момент».
  - **Таб «Меню»:** от **одной генерации** (1 позиция — то же качество, что и пакет, но на 1 блюдо) до пакетов позиций (напр. 5/10/20/50). То есть входной порог — 1 фото, дальше пакеты.
  - **Таб «Поток»:** тарифы-подписки (ежедневный контент, дешевле за кадр).
- **Акцент на рекомендуемом тарифе:** самый выгодный нам пакет/подписку выделяем сильным акцентом (бейдж «Выгоднее всего» / приподнятая карточка / акцентная рамка), чтобы направлять выбор — как в большинстве современных сервисов.
- **Бесплатная первая генерация:** при регистрации в боте даётся 1 генерация **по режиму «Меню»** бесплатно как тест (в само меню она не сохраняется). Это и есть красная нить «первое фото бесплатно» — упомянуть у CTA и в FAQ.
- **Сообщение:** «Выберите задачу — оба режима доступны всем, можно комбинировать».
- **Действие:** CTA «Начать бесплатно» → в бота.
- **Мир:** master (нейтральный — предложение одно для всех сегментов).
- **Заметки / состояния:**
  - Режимы объясняются здесь и только здесь (не в сегментах) — развязка осей.
  - «Меню» не вываливает все карточки как равнозначную стену — визуальная иерархия: одна карточка-рекомендация в акценте, «одна генерация» как лёгкий вход, остальное считывается вторично.
  - Мелкий серый текст-заверение поднять до читаемого контраста (WCAG AA).

### 7. FAQ / возражения
- **Цель:** Снятие последних возражений перед действием.
- **Содержание:** аккордеон. **Наполнение сейчас — плейсхолдер (рандомные вопросы), тексты меняются позже.** Ориентиры-темы для верстки-рыбы: «нейросеть не исказит блюдо», «не будет ИИ-глянца», вотермарки, оплата/отмена, скорость, что такое бесплатная первая генерация.
- **Сообщение:** «Рисков нет — вы контролируете результат».
- **Действие:** —
- **Мир:** master.
- **Заметки:** семантика — нативные `<details>/<summary>` вместо checkbox-хака (лучше для скринридеров).

### 8. Финальный CTA → бот
- **Цель:** Действие.
- **Содержание:** спокойный низко-давящий заголовок, «первое фото — бесплатно, дальше решите сами», единственная **реальная внешняя ссылка** на Telegram.
- **Сообщение:** «Попробуйте — это ничего не стоит».
- **Действие:** primary-CTA → `BOT_URL` (target="_blank").
- **Мир:** master (акцентный, но primary-цвет бренда обязателен — не терять фирменный акцент в тёмной теме).
- **Заметки:** это финал peak-end — тон рассчитан на осторожного малого бизнеса.

### 9. Footer
- **Цель:** доверие/юр.
- **Содержание:** контакты, соцсети, оферта/политика, © .
- **Мир:** master.

## 6. Сквозные элементы воронки

### 6.1 Единый CTA
- Одна формулировка для реального перехода в бота (напр. «Попробовать бесплатно» / «Открыть бота»).
- Внутренние кнопки-якоря НЕ называть «Открыть бота» — давать честную подпись действия («Посмотреть как», «Смотреть тарифы»). Чинит проблему «5 одинаковых кнопок с разным поведением».

### 6.2 Красная нить «первое фото бесплатно»
- Повторяется в каждой точке решения: hero → сегмент → тарифы → финал. Главный сниматель трения.

### 6.3 Навигация
- Sticky-хедер с якорями на общие секции (Как работает · Тарифы · Вопросы) + возможность прыгнуть к своему сегменту из развилки.

### 6.4 Анимации — только на смысле
- Reveal трансформации (§2), морфинг сегмента (§4), появление цены (§6). Не «анимация ради анимации».
- Везде уважать `prefers-reduced-motion`.

### 6.5 Фото-центричность — с границей
Контент строится вокруг фотографий: у продукта фото — это сам аргумент («show, don't tell» = питч продукта, а настоящие выходы бота — материал, который клон на ИИ не подделает). Но применяется **не равномерно, а по половинам воронки**:
- **Верхняя половина (расхождение — эмоция/пруф) — фото ведёт:** §1 (визуал-якорь + сильный заголовок), §2 (full-bleed до/после), §3 (карточки-фото), §4 (боль/обещание сегмента картинкой, не абзацами), §5 (результат-фото в центре, текст = подписи шагов).
- **Нижняя половина (схождение — логика/решение) — текст ведёт:** §6 тарифы (числа и иерархия карточек), §7 FAQ (возражения вербальны), заверения. Фото здесь максимум поддержка.

Формулировка не «минимум текста везде», а **«каждое слово оправдано»**: фото несёт то, что доказуемо картинкой; слова несут только обещание, цену, заверение, возражение.

Два условия именно для нас:
- **Ставка на качество ассетов растёт** — плейсхолдеры при фото-центричности бьют вдвое сильнее (см. §8).
- **Два сегмента — противоположная эстетика фото:** заведение = глянцевый студийный кадр; дом = НЕ глянец, а настоящее блюдо с хорошим светом (совпадает с принципом честности бренда). «Максимум визуала» — по-разному в двух подтемах.

Технические следствия (обязательны): производительность (responsive/lazy-load, AVIF/WebP, предзагрузка только дефолтных кадров — особенно 60 комбинаций §5) и доступность (текст живой, не запечён в картинку; alt; контраст надписей поверх фото).

## 7. Что структура диктует дизайн-системе

Структура задаёт архитектуру токенов и компонентов:

- **Темы:** 1 master-тема (core-токены) + 2 сегмент-подтемы (`studio-dark`, `home-warm`) как наборы **семантических оверрайдов** (фон, поверхность, акцент, текст). Никаких «двух полных сайтов».
- **Токены:** цвет (core + акцент бренда, обязателен во всех темах, включая тёмную), типографика (дисплейная + текстовая шкала), радиусы, тени, spacing-ритм, длительности/кривые анимаций (для морфинга и reveal).
- **Ключевые компоненты (из секций):** Header/Nav, Button (primary/ghost, единый CTA), BeforeAfter (интерактивный + fallback), SegmentSwitcher (морфинг-контроллер), SegmentBlock (каркас под обе подтемы), StepList (3 шага), StyleGallery (курируемая + «все»), PricingTabs (Меню/Поток, единый), FAQ (details/summary), CTASection, Footer.
- **Паттерны:** «морфинг мира на месте», «эхо акцента» в общие блоки, «reveal при скролле», «reassurance-плашка у точек решения».
- **Состояния:** default/hover/focus/active для интерактива; no-JS fallback для BeforeAfter; reduced-motion-варианты; мобильные раскладки (thumb-zone для CTA, компактные галереи).

## 8. Контент, который готовится отдельно (позже)

- **§2:** одна реально контрастная пара «до/после» (телефонный снимок ↔ студийный результат).
- **§5 конфигуратор:** предгенерированный набор комбинаций «блюдо × стиль × локация» (напр. 5 × 4 × 3 = 60 готовых изображений) + сами «ДО»-снимки шага 1. Числа наборов зафиксировать ДО генерации.
- **§1 hero:** ассет мокапа телефона с Telegram-ботом (или альтернативный акцентный герой-элемент).
- **§3 развилка:** два сегмент-конгруэнтных фото для карточек (ресторанная подача / домашний торт).
- Финальные тексты (в т.ч. реальный FAQ), цены/тарифы, фирменные цвета/шрифты.

## 9. Решённые вопросы

- **Дефолт развилки** — «Заведение» (кафе/рестораны) preselected.
- **Третий вход под кейтеринг** — не нужен; растворяется в двух вариантах по задаче.
- **Глубина «эха акцента»** — акцентные элементы (кнопки, тексты, ссылки) под сегмент; структура/фоны нейтральны (см. §4).
- **Дизайн-система и вёрстка** — выполняются в отдельных сессиях; этот документ — их вход.
